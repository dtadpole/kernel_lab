# FA4 2-Consumer-WG Attempt — Failed (Reverted)

## Objective

Add a second consumer warp group (tile_m=128) to double MMA parallelism.
Expected: ~600+ TFLOPS (near cuDNN). Actual: unstable, hangs, worse performance.

## What Was Built

- 384 threads: 1 producer WG + 2 consumer WGs (each processes m=64)
- `setmaxnreg.dec 24` (producer) / `setmaxnreg.inc 240` (consumer)
- Warp scheduler barriers (bar.sync/bar.arrive) between consumer WGs
- BLOCK_Q=128, BLOCK_KV=128, DIM=128
- SMEM: Q=32KB + K=64KB + V=64KB = 160KB (within 228KB)
- 168 registers, 0 spills, 16 hardware barriers

## Results

| Config | 1-WG (stable) | 2-WG (attempt) | Status |
|--------|--------------|----------------|--------|
| nc-b8-s4096 | 436 | 504 (1st launch only) | Hangs on some configs |
| nc-b4-s8192 | 457 | 520 (1st launch only) | Performance varies |
| nc-b2-s16384 | 471 | 530 (1st launch only) | TIMEOUT on repeated launches |
| c-b8-s4096 | 357 | 412 (no sched barriers) | Hangs with sched barriers |
| c-b4-s8192 | 388 | HANG | |
| c-b2-s16384 | 412 | HANG | |

## Root Cause of Failures

1. **Warp scheduler barriers (bar.sync/bar.arrive)** cause deadlocks with
   causal masking where max_kv_iter varies across Q blocks (e.g., block 0
   has max_kv_iter=1, block 31 has max_kv_iter=32).

2. **Without scheduler barriers**, the 2 WGs interfere — performance drops
   to ~236 TFLOPS (worse than 1-WG!) because WGs race on shared K/V buffers.

3. **Repeated kernel launches** with bar.sync sometimes hang even for
   noncausal, suggesting barrier state corruption across CTA scheduling waves.

## Learning

- The CuTe DSL uses sophisticated barrier patterns (NamedBarrierFwd enum,
  per-WG barrier IDs, canonical_warp_group_idx) that ensure correct pairing.
- Simple bar.sync with 256 threads across 2 WGs is fragile.
- Need to study CuTe DSL's exact barrier protocol more carefully, especially
  the mma_init pattern and how it handles first vs middle vs last iterations.
- The 1-WG kernel at 436-471 TFLOPS is the stable baseline for now.

## Decision

Reverted to 1-WG kernel. The 2-WG approach needs more careful implementation
of the inter-WG coordination protocol before it can be production-ready.
