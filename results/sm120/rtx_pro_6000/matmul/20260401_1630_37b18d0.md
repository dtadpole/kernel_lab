## Implementation Techniques

### Reference: CuTe DSL (`conf/fixtures/matmul/cute_gemm.py`)

The reference uses NVIDIA's CuTe DSL (CUTLASS Python bindings) with MLIR JIT compilation.

**Architecture:**
- **Tile shape**: 128×128×64 (M×N×K)
- **Thread count**: 160 (5 warps: 1 DMA + 4 MMA)
- **Pipeline stages**: 3
- **Register budget**: ~232 (MMA warps), ~40 (DMA warp) via `setmaxregister`

**Key techniques:**
1. **TMA (Tensor Memory Accelerator)** — `cp.async.bulk.tensor.2d` for global→SMEM loads. Hardware-driven 2D tile copies with zero thread involvement. Uses `CopyBulkTensorTileG2SOp`.
2. **Warp specialization** — 1 DMA warp (producer) handles all TMA loads; 4 MMA warps (consumers) do compute. Decouples memory and compute for better overlap.
3. **PipelineTmaAsync** — Multi-stage (3-stage) software pipeline with `mbarrier` synchronization. Producer acquires/commits stages; consumers wait/release.
4. **ldmatrix S2R** — `ldmatrix.sync.aligned.x4.m8n8` for shared→register data movement, directly feeding MMA operands.
5. **mma.sync** — `mma.sync.aligned.m16n8k16.f32.bf16.bf16.f32` Tensor Core operation with 2×2×1 warp layout (4 warps × 32×16 per warp = 128×128 tile).
6. **Persistent tile scheduling** — `StaticPersistentTileScheduler` assigns multiple output tiles per CTA via work loop. Max 188 CTAs (matching SM count).
7. **Zero-copy input** — `from_dlpack(B.t())` creates transposed view without copying data. B layout: (N,K) N-major from PyTorch's column-major view.
8. **BF16 epilogue** — FP32 accumulator → BF16 output conversion for reduced memory traffic.

### Generated: Hand-Written CUDA (`data/generated/sm120/matmul/generated.cu`)

The generated kernel is hand-written PTX-level CUDA with two size-specialized variants.

**Architecture (Big kernel, M,N ≥ 2048):**
- **Tile shape**: 256×128×32 (M×N×K)
- **Thread count**: 256 (8 warps: 2M×4N layout)
- **Pipeline stages**: 3
- **Registers**: 191 (0 spills)

**Architecture (Small kernel, M,N < 2048):**
- **Tile shape**: 128×64×32 (M×N×K)
- **Thread count**: 128 (4 warps: 2M×2N layout)
- **Pipeline stages**: 4
- **Registers**: 134 (0 spills)

**Key techniques:**
1. **TMA via inline PTX** — `cp.async.bulk.tensor.2d.shared::cta.global.tile` with explicit `CUtensorMap` encoding via driver API (`cuTensorMapEncodeTiled`).
2. **mbarrier pipeline** — Manual `mbarrier.init` / `mbarrier.arrive.expect_tx` / `mbarrier.try_wait.parity` for producer-consumer synchronization. Phase-based completion tracking.
3. **ldmatrix** — `ldmatrix.sync.aligned.x4.m8n8.shared.b16` for A tiles; `ldmatrix.sync.aligned.x2.trans.m8n8.shared.b16` for B tiles (transposed load).
4. **mma.sync** — Same `mma.sync.aligned.m16n8k16.row.col.f32.bf16.bf16.f32` as reference.
5. **SMEM swizzle** — A uses 64B swizzle (`CU_TENSOR_MAP_SWIZZLE_64B`), B uses 128B swizzle (`CU_TENSOR_MAP_SWIZZLE_128B`) to avoid bank conflicts.
6. **Interleaved load + compute** — Big kernel interleaves A-tile loads with MMA in groups of 2, overlapping ldmatrix latency with compute.
7. **Persistent grid** — `min(totalTiles, numSMs)` blocks with tile loop. 4×4 super-tile ordering for L2 locality when grid is large enough.
8. **Multi-tile dispatch** — Size-based kernel selection at launch time. Big kernel has wider tiles (256×128) for better compute density at large sizes; small kernel has more pipeline stages (4) for better latency hiding at small sizes.
9. **BF16 epilogue** — Direct `__floats2bfloat162_rn` vectorized stores (bfloat162 = 2 elements per store).
10. **`__launch_bounds__(N, 1)`** — Explicit occupancy hint, 1 block per SM for maximum register budget.

### cuBLAS: Vendor Baseline (`conf/fixtures/matmul/cudnn.py`)

- **Implementation**: `torch.mm()` → cuBLAS `cublasGemmEx`
- **Details**: Closed-source. BF16 inputs, FP32 internal accumulation. cuBLAS auto-selects the best Tensor Core kernel for the GPU architecture via internal heuristics.
- **Strengths**: Extremely well-tuned for large GEMM sizes. Benefits from architecture-specific tuning that may use features not available in public APIs.

## Key Observations

1. **Generated beats cuBLAS at small/medium sizes** (256–2048): up to 2.48x at 256×256. The generated kernel's multi-tile dispatch with size-specific tile shapes gives it an advantage where cuBLAS's large-tile heuristics have too much overhead.

2. **Generated matches cuBLAS at large sizes** (4096–8192): within 1% at 4096, tied at 8192. Both achieve ~88% of peak, suggesting the problem is truly compute-bound at these sizes.

3. **Reference is slower due to Python/MLIR overhead**: The CuTe DSL kernel has ~17μs per-call overhead from the Python→MLIR→CUDA dispatch path. This dominates at small sizes (256–512) where kernel execution itself is <10μs. At 8192×8192 the reference reaches 80% peak — the GPU kernel itself is competitive, but the dispatch overhead caps it below the other two.

4. **No register spills in generated kernel**: 191 registers (big) and 134 registers (small), both with 0 spills. The reference uses up to 232 registers in MMA warps.
