## Benchmark Configuration

- **Host**: devvm8490, GPU index 4
- **Measurement**: CUDA event timing, 5 warmup + 20 timed trials per config, median reported
- **Correctness**: Verified via cuda_exec evaluate pipeline (6/6 configs passed, atol=1e-2, rtol=1e-2)
- **All configs**: H=16, D=128, total_tokens=B×S=32768

## Implementations

1. **FA4 CuTe DSL** (`cutedsl.py`): `flash_attn.cute.flash_attn_func` — SM80 base class path on SM90
2. **cuDNN** (`cudnn.py`): `torch.nn.functional.scaled_dot_product_attention` with `CUDNN_ATTENTION` backend forced
3. **Generated** (`generated.cu`): Hand-written warp-specialized kernel, 5 warps (1 DMA + 4 MMA), BLOCK_Q=128, BLOCK_KV=64, DIM=128, `mma.sync.m16n8k16`, `cp.async`, 80KB SMEM

## Analysis

- **FA4 CuTe DSL vs cuDNN**: Near-parity overall. cuDNN leads on causal configs (554–610 vs 543–599 TFLOPS), FA4 leads on non-causal (649–681 vs 599–661 TFLOPS).
- **Generated kernel**: 100–115 TFLOPS, ~17% of FA4/cuDNN. Uses SM80-compatible `mma.sync.m16n8k16` path; migrating to SM90-native `wgmma` + TMA would be the primary optimization target.
- **H100 BF16 utilization**: FA4/cuDNN achieve 55–69% of peak 989.5 TFLOPS; generated kernel at 10–12%.
