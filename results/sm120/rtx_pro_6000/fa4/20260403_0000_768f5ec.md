## What Changed

### Early K_EMPTY Signal (commit `768f5ec`)

Moved the `bar_arrive(BAR_K_EMPTY)` from **after** the entire QK+softmax loop to **inside** the `mma_id_q` loop, right after the last QK d-loop iteration. This releases the K shared memory buffer ~500 cycles earlier (before softmax scale, causal mask, online softmax, and P packing), giving the DMA warp a head start on loading K_{i+1}.

**Before** (baseline `a54a17d`):
```
for mma_id_q = 0..1:
    QK d-loop        // reads K_smem
    softmax scale
    causal mask
    online softmax
    pack P
bar_arrive(K_EMPTY)  // K released here — after ALL softmax work
```

**After** (optimized `768f5ec`):
```
for mma_id_q = 0..1:
    QK d-loop        // reads K_smem
    if last mma_id_q:
        bar_arrive(K_EMPTY)  // K released here — immediately after last K read
    softmax scale
    causal mask
    online softmax
    pack P
```

**Why it helps**: The DMA warp waits on `BAR_K_EMPTY` before loading K_{i+1}. By signaling earlier, the DMA warp starts loading ~500 cycles sooner per KV iteration. Over 64 iterations (seq_len=4096), this saves ~32,000 cycles total.

## Implementation Techniques (Current Kernel)

### Architecture
- **Tile shape**: BLOCK_Q=128, BLOCK_KV=64, DIM=128
- **Thread count**: 160 (5 warps: 1 DMA + 4 MMA)
- **Warp specialization**: Warp 0 loads K/V via `cp.async.cg`; Warps 1-4 compute via `mma.sync.m16n8k16`
- **SMEM layout**: Q (32KB persistent) + K (32KB double-buffered) + V (16KB single-buffered) = 80KB

### Key Techniques
1. **Warp specialization** with named barriers (`bar.sync`/`bar.arrive`) for DMA/MMA synchronization
2. **K double-buffer** — DMA prefetches K_{i+1} while MMA computes with K_i
3. **Early K_EMPTY** — release K buffer right after QK (before softmax), giving DMA ~500 cycles head start
4. **ldmatrix + MMA pipeline** — Q fragments prefetched via `ldmatrix.x4` for next d-step during current MMA
5. **V fragment pipeline** — V fragments prefetched via `ldmatrix.x2.trans` for next (kv,d) step during PV MMA
6. **Online softmax** — log2-based with butterfly warp reduction (`__shfl_xor_sync`)
7. **Causal tile-skip** — tile-level early-exit when all KV positions ≤ all Q positions
8. **SMEM swizzle** — XOR-based for bank-conflict-free `ldmatrix` access
9. **`__launch_bounds__(160, 1)`** — forces full 255-register budget (1 block/SM)

## Optimization Attempts (This Session)

### Attempt 1: Cooperative Loading (No Warp Specialization)

**Hypothesis**: 66% barrier stalls come from the DMA warp being too slow (32 threads). Using all 128 threads for cooperative K/V loading would be 4x faster.

**Change**: Removed warp specialization entirely. All 128 threads load K/V cooperatively via `cp.async`, then all 4 warps compute QK/PV. Double-buffered K with `cp.async.wait_group` for pipelining. No named barriers, just `__syncthreads`.

**Result**: **-14% regression** (0.78x vs cuDNN, down from 0.92x).

**Root cause**: Lost DMA/MMA overlap. In the warp-specialized version, DMA loads V while MMA computes QK (overlap). In cooperative loading, loading and computing are serial: load K → compute QK → load V → compute PV. Even though loading is 4x faster per tile, the total serial time is worse than overlapped time.

**Bug discovered**: `cp.async.wait_group` completes groups in FIFO order (oldest first). When K prefetch was committed before V, `wait_group 1` completed K (oldest) but left V pending — causing correctness failures on causal configs. Fix: commit V before K prefetch.

**Lesson**: For compute-bound kernels like flash attention, DMA/MMA overlap is more valuable than faster loading speed. The 47% stall_wait is the cost of overlap, but it's still better than no overlap.

### Attempt 2: Pipelined K+V Loading (Wait Both Barriers First)

**Hypothesis**: Issue K and V `cp.async` back-to-back so the memory controller pipelines both.

**Change**: DMA warp waits for both `K_EMPTY` and `V_EMPTY` at the start of each iteration, then issues K and V loads sequentially without waiting for K completion between them.

**Result**: **-10% regression** (0.83x vs cuDNN).

**Root cause**: Waiting for `V_EMPTY` at the start delays the entire DMA operation. In the original, K loading starts as soon as `K_EMPTY` arrives (early, after QK). `V_EMPTY` arrives later (after PV). By waiting for both upfront, the DMA warp is idle during PV.

### Attempt 3: Pipelined K+V Loading (Separate Barriers)

**Hypothesis**: Keep original barrier pattern (K_EMPTY before K, V_EMPTY before V) but don't wait for K completion before starting V loads.

**Change**: DMA warp issues K `cp.async`, then waits for `V_EMPTY`, issues V `cp.async`, then waits for K (`wait_group 1`), then V (`wait_group 0`).

**Result**: **±0%** (0.91–0.92x vs cuDNN, same as baseline).

**Root cause**: K and V loads share DRAM bandwidth. Overlapping the loads saves only the latency (~1000 cycles) but not the bandwidth time (~4000 cycles each), because both use the same memory bus. Net saving is too small to measure.

### Attempt 4: Early K_EMPTY Signal (KEPT)

**Hypothesis**: Release K buffer right after QK loop (before softmax), giving DMA ~500 extra cycles to start K_{i+1}.

**Change**: Move `bar_arrive(BAR_K_EMPTY)` from after the entire QK+softmax loop to inside the `mma_id_q` loop, after the last QK d-loop.

**Result**: **+1% improvement** across all 6 configs. Consistent 0.7–1.2% latency reduction.

**Root cause**: DMA warp gets ~500 cycles earlier start on K_{i+1} loading per KV iteration. Over many iterations, this compounds.

## Remaining Gap Analysis

The generated kernel achieves **0.91–0.94x vs cuDNN** (6–9% gap). Root causes:

1. **cp.async vs TMA**: cuDNN likely uses TMA (`cp.async.bulk.tensor`) for hardware-accelerated DMA. TMA loads entire tiles in a single instruction with zero thread overhead, vs `cp.async.cg` which requires 32 iterations per thread for address computation + 16-byte loads.

2. **Named barriers vs mbarrier**: cuDNN uses `mbarrier` (shared memory barriers) which are more efficient than named `bar.sync`/`bar.arrive` for producer-consumer synchronization.

3. **Instruction scheduling**: cuDNN has proprietary compiler optimizations for instruction interleaving between loads and compute.

### SM120 Constraints (Verified This Session)

| Feature | Available on SM120? | Notes |
|---------|-------------------|-------|
| `mma.sync.m16n8k16` | Yes | Current instruction set |
| `wgmma` (warp group MMA) | **No** | Requires `sm_90a` (Hopper) |
| `tcgen05.mma` (5th gen TC) | **No** | Requires `sm_100a` (B200), needs TMEM |
| TMA (`cp.async.bulk.tensor`) | Yes | Available but requires host-side tensor maps |
| `mbarrier` | Yes | More efficient than named barriers |

### Path to Closing the Gap

The only viable path to significantly reduce the 8% gap is **TMA migration**:
1. Host-side tensor map creation (`cuTensorMapEncodeTiled` via CUDA Driver API)
2. `mbarrier`-based producer/consumer pipeline (replaces named barriers)
3. Kernel restructure for single-thread TMA initiation
4. Reference implementation: CUTLASS Blackwell GeForce `dense_gemm.py` shows the pattern
