# FA4 TMA + Split-DIM SMEM — 405–507 TFLOPS

## Hardware

- **GPU:** NVIDIA H100 SXM5 80GB HBM3
- **Architecture:** SM 9.0a (Hopper)
- **SMs:** 132
- **Peak BF16 Tensor Core:** 989.5 TFLOPS (dense)
- **HBM3 Bandwidth:** 3,352 GB/s
- **SMEM per SM:** 228 KB
- **Host:** devvm8490.cco0.facebook.com (h8_3), GPU 4
- **CUDA Toolkit:** 12.8

## Objective

Replace cp.async software loads with TMA (Tensor Memory Accelerator) hardware
loads to eliminate ~350 address computation instructions (IMAD/IADD3/LOP3) per
KV iteration. This was the dominant bottleneck identified via SASS analysis.

## Approach: Split-DIM TMA Layout

TMA with SWIZZLE_128B limits boxDim[0] to 64 for BF16 (128 bytes / 2 bytes).
DIM=128 requires two TMA loads per tile: cols 0-63 (lo) and cols 64-127 (hi).

**Key insight:** TMA writes groups contiguously (1024B each), but the previous
interleaved layout expected 2 atoms per group (2048B). Split lo/hi halves into
separate contiguous SMEM sections to match TMA's output pattern.

## Architecture

| Parameter | Value |
|-----------|-------|
| Threads | 128 (1 warp group, 4 warps) |
| tile_m (BLOCK_Q) | 64 |
| tile_n (BLOCK_KV) | 64 |
| DIM (head_dim) | 128 (split: 64 lo + 64 hi) |
| Data loading | TMA (`cp.async.bulk.tensor.2d`) |
| QK WGMMA | `m64n64k16.f32.bf16.bf16` (SS mode) |
| PV WGMMA | `m64n64k16.f32.bf16.bf16` (RS mode, 2 calls: V_lo→O_lo, V_hi→O_hi) |
| Pipeline | 2-stage K (double-buffered), dedicated V buffer |
| Sync | mbarrier (TMA auto-signals on completion) |
| SMEM | 8 sections × 8KB = 64KB (Q_lo, Q_hi, K0_lo, K0_hi, K1_lo, K1_hi, V_lo, V_hi) |
| WGMMA stride | 1024B (1 atom per group) |
| Registers | ~170/thread, 0 spills |
| Occupancy | 1 block/SM |

### TMA Descriptor Setup

- Per-(batch, head) descriptors via `cuTensorMapEncodeTiled` (dlopen libcuda.so.1)
- 2D tensor: globalDim = {64, S}, globalStrides = {H×D×2 bytes}
- boxDim = {64, BLOCK} with `CU_TENSOR_MAP_SWIZZLE_128B`
- Two TMA loads per tile: coord0=0 (lo) and coord0=64 (hi)
- B×H descriptor sets allocated in GPU global memory

### QK K-stepping

- ks 0-3: use Q_lo/K_lo descriptors, advance ks×2
- ks 4-7: use Q_hi/K_hi descriptors, advance (ks-4)×2

### PV K-stepping (tnspB=1)

- Two m64n64k16 RS calls per K-step (V_lo→O_lo, V_hi→O_hi)
- V_KS_ADVANCE = 2 × 1024 / 16 = 128 per K-step (16 rows = 2 groups)
- LBO = 1 (single atom per row in split layout)

## What Worked

1. **TMA eliminates address computation overhead** — SASS showed 352 integer
   instructions (IMAD/IADD3/LOP3) for cp.async + swizzle address calculation.
   TMA replaces all of these with 2 hardware-accelerated bulk tensor loads per
   tile (lo + hi halves). Net: ~11% throughput improvement.

2. **Split-DIM layout matches TMA's contiguous group placement** — TMA with
   SWIZZLE_128B writes 1024B groups contiguously. Splitting DIM=128 into two
   64-col halves stored separately avoids the atom interleaving mismatch.

3. **mbarrier replaces __syncthreads** — TMA auto-signals mbarrier on
   completion. Lighter weight than cp.async.wait_all + __syncthreads.

4. **Per-(batch,head) TMA descriptors** — cuTensorMapEncodeTiled creates
   descriptors on host, cudaMemcpy to device. Each block loads its descriptor
   from the array. Handles the strided (B,S,H,D) layout correctly.

## What Didn't Work (Failed Attempts)

1. **boxDim[0]=128 with SWIZZLE_128B** — INVALID for BF16.
   Max boxDim[0] = 128B / sizeof(BF16) = 64. Caused silent data corruption
   (output all zeros). Discovered after 2 failed TMA iterations.

2. **Interleaved SMEM layout with TMA** — TMA writes groups contiguously but
   interleaved layout expects 2 atoms per group. Data from group 1 overwrites
   atom 1 of group 0. Split-DIM layout fixes this.

3. **Warp specialization with cp.async (Phase 2)** — 32-thread producer too
   slow for cp.async loads (4× fewer threads = 4× slower loads). TMA solves
   this since only 1 thread issues TMA and hardware handles bandwidth.

## Remaining Gap to CuTe DSL (~1.3–1.5×)

| Gap Source | Estimated Impact | Fix |
|-----------|-----------------|-----|
| 1 WG vs 2 WGs | ~1.2× | 2 MMA warp groups with intra-WG overlap |
| tile_m=64 vs 128 | ~1.1× | Larger tiles (requires 2 WGs) |
| No QK/PV overlap | ~1.1× | Issue QK[n] and PV[n-1] as separate commit groups |
| Sequential lo/hi PV | ~1.05× | Fuse into single m64n128k16 (needs interleaved V layout) |

## Correctness

- Max error: 32 (BF16 precision limit at scale ~8000)
- Cols 120-127: 0 error (previously 960 with wrong LBO)
- Test: S=64, H=1 non-causal arange data vs PyTorch SDPA reference
