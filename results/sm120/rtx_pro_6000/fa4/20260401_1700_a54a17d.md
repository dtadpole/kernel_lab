## Implementation Techniques

### Reference: CuTe DSL (`flash_attn.cute.flash_attn_func`)

FA4 CuTe DSL is NVIDIA's official FlashAttention-4 implementation using the CuTe DSL (CUTLASS Python bindings) with MLIR JIT compilation for SM120.

**Key characteristics:**
- Uses **wgmma** (warp group MMA) — SM120 native tensor core instruction
- Uses **TMA** (Tensor Memory Accelerator) for global→SMEM loads
- Requires 3 patches on SM120 (see `conf/fixtures/fa4/DESIGN.md`)
- JIT compilation: first call compiles ~100s, cached thereafter
- Non-causal achieves 83–88% peak (417–443 TFLOPS) — fastest backend

### Generated: Hand-Written CUDA (`data/generated/sm120/fa4/generated.cu`)

Warp-specialized flash attention kernel using inline PTX assembly.

**Architecture:**
- **Tile shape**: BLOCK_Q=128, BLOCK_KV=64, DIM=128
- **Thread count**: 160 (5 warps: 1 DMA + 4 MMA)
- **Registers**: 255 (12B spill store, 28B spill load in mma_warp_fn)
- **SMEM**: 80 KB (Q: 32KB persistent + K: 32KB double-buf + V: 16KB single-buf)
- **Occupancy**: 10.4% (1 block/SM, limited by SMEM)

**Key techniques:**
1. **Warp specialization** — Warp 0 (DMA) loads K/V via `cp.async.cg.shared.global`; Warps 1-4 (MMA) compute QK + softmax + PV. Named barriers (`bar.sync`/`bar.arrive`) for DMA/MMA synchronization.
2. **K double-buffer** — Two SMEM slots for K tiles, allowing DMA to prefetch K[i+1] while MMA consumes K[i].
3. **ldmatrix + mma pipeline** — Q fragments prefetched via `ldmatrix.sync.aligned.x4` for next d-step while current MMA executes. V fragments similarly pipelined with `ldmatrix.x2.trans`.
4. **mma.sync.aligned.m16n8k16** — BF16 inputs, FP32 accumulation. 4 MMA warps × 32 Q rows = 128 Q rows per block.
5. **Online softmax** — Log2-based softmax with butterfly warp reduction. Rescaling with `exp2f` for numerical stability.
6. **Causal tile-skip** — Tile-level check skips the per-element masking loop when all KV positions ≤ all Q positions for the current warp/mma_id_q.
7. **SMEM swizzle** — XOR-based swizzle pattern for bank-conflict-free ldmatrix access.
8. **`__launch_bounds__(160, 1)`** — Forces compiler to use full 255 registers (1 block/SM). Critical: without this, compiler reduces to 168 regs causing 20–30% regression.

### cuDNN SDPA: Vendor Baseline (`reference.py` fallback)

- **Implementation**: `torch.nn.functional.scaled_dot_product_attention` → cuDNN flash attention
- **Details**: Closed-source. BF16 inputs. Auto-selected by PyTorch when `SDPBackend.CUDNN_ATTENTION` is available.
- **Performance**: Consistent 72–76% peak across all configs. Faster than CuTe DSL on causal (likely uses proprietary triangular work distribution).

### Roofline Position

- **Per-tile operational intensity**: 128 FLOP/byte (GMEM→SMEM, no L2)
- **Algorithm OI with L2 reuse**: 1024–8192 FLOP/byte
- **Regime**: Deeply compute-bound (L2 absorbs K/V reuse)
- **Bottleneck**: Tensor core utilization (60%) limited by barrier sync overhead (66% of stall time)

## Improvement Opportunities

1. **Reduce barrier overhead** (66% of stalls): V double-buffer with pipelined barriers could overlap DMA V load with MMA PV compute. Need careful barrier redesign to avoid the complexity that hurt the previous attempt.

2. **Upgrade to wgmma** (SM120 native): CuTe DSL's 83–88% utilization on non-causal shows what wgmma can achieve. Our mma.sync.m16n8k16 tops out at ~75%. Requires fundamental instruction-level rewrite.

3. **TMA for global loads**: Replace cp.async with TMA to reduce DMA warp overhead and free the DMA warp for other work. TMA handles 2D tile copies in hardware.

4. **Causal work distribution**: cuDNN is faster on causal despite CuTe DSL being faster on non-causal, suggesting proprietary triangular block scheduling. Could implement interleaved block assignment (early + late blocks on same SM).
