# FA4 BLOCK_KV 64→128 — 407–537 TFLOPS (+1–4%)

## Hardware

- **GPU:** NVIDIA H100 SXM5 96GB HBM3
- **Architecture:** SM 9.0a (Hopper)
- **SMs:** 132
- **Peak BF16 Tensor Core:** 800 TFLOPS (650W TDP)
- **HBM3 Bandwidth:** 2,447 GB/s
- **Host:** devvm8490.cco0.facebook.com (h8_3), GPU 4
- **CUDA Toolkit:** 12.8

## Objective

Halve KV loop iterations by doubling BLOCK_KV from 64 to 128. This directly
reduces the 48% synchronization overhead (wgmma.wait + mbarrier + warpgroup_arrive)
identified by NCU profiling.

## Approach

1. **New WGMMA wrapper**: `wgmma_m64n128k16_f32_bf16` (SS mode, 64 output registers)
   for QK GEMM. Produces 64×128 output in a single call per k-step.
2. **SMEM layout**: Increased from 64KB to 112KB per block:
   - Q: 2×8KB = 16KB (unchanged)
   - K (double-buffered): 4×16KB = 64KB (was 4×8KB = 32KB)
   - V: 2×16KB = 32KB (was 2×8KB = 16KB)
3. **TMA descriptors**: K and V boxDim changed from {64, 64} to {64, 128}.
4. **QK GEMM**: m64n128k16 SS mode (8 k-steps for DIM=128 split). S_acc[64].
5. **Softmax**: Processes 128 KV positions per iteration (32 values per thread,
   4 threads per row → 128 columns).
6. **PV GEMM**: RS mode with 8 k-steps per half (was 4), matching BLOCK_KV=128.
7. **Registers**: 128→168 per thread, 0 spills.

## NCU Profiling (before, BLOCK_KV=64, nc-b2-s16384)

Key metrics that motivated this optimization:

| Metric | Value |
|--------|-------|
| Tensor core utilization | 56.3% |
| Compute SM throughput | 56.3% |
| Memory throughput | 75.1% (L2-dominated, DRAM 2.1%) |
| Achieved occupancy | 18.5% (3 blocks/SM) |
| Warp stall: wait (wgmma.wait) | 24.0% |
| Warp stall: barrier (mbarrier) | 18.9% |
| Warp stall: warpgroup_arrive | 5.4% |
| **Total sync overhead** | **48.3%** |

## What Worked

BLOCK_KV=128 provides consistent 1–4% improvement across all configs:
- Halves KV loop iterations → fewer barrier waits and WGMMA sync points
- m64n128k16 halves QK WGMMA call count per iteration (same per k-step)
- Larger softmax tile amortizes scalar overhead per KV position
- Improvement scales with sequence length (4% for S=16384 vs 1% for S=4096)

## Why Improvement Is Modest (vs 1.23× pre-TMA)

The pre-TMA kernel gained 1.23× from BLOCK_KV=128 because cp.async had massive
per-iteration address computation overhead (~21% IMAD+LOP3 instructions). TMA
eliminated that overhead, so the remaining per-iteration cost is lower.

The dominant remaining bottleneck (48% sync overhead) is only partially addressed:
- Halving iterations halves sync point count → ~24% savings on 48% = ~12% reduction
- But SMEM increase (64KB→112KB) drops occupancy from 3→2 blocks/SM
- Fewer warps = less latency hiding during sync points → partially offsets savings
- Net effect: ~4% improvement instead of the expected ~12%

| Registers/thread | 128 | 168 |

## Remaining Gap to cuDNN (~1.15–1.2×)

| Gap Source | Impact | Fix |
|-----------|--------|-----|
| Low occupancy (2 blocks) | ~1.05× | Reduce SMEM via K/V buffer sharing or single-buffer K |
| QK/PV serial execution | ~1.1× | Overlap PV[i] with K[i+1] TMA via wgmma_wait_group<1> |
| 1 WG vs 2 WGs | ~1.1× | Add second consumer WG (requires correct barrier protocol) |
