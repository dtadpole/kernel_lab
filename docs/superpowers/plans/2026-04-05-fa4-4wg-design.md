# FA4 4-WG Architecture Design — SM90 H100

## Goal
Implement 4 warp group Flash Attention on SM90 to increase scheduler warp utilization
(from 0.59 → ~0.8 eligible warps/scheduler) and close the remaining 1-4% gap to FA4 CuTe DSL.

## Current Best: 3-WG V3 + IS_CAUSAL Template
- 384 threads = 3 WGs (Producer + 2 Consumer)
- 168 regs, 0 spills, 16 barriers
- Peak: 548 TF causal b8s4096 (0.99× cuDNN), 605 TF causal b2s16384 (1.11× cuDNN)

## Blocking Issue: Compiler warpgroup.arrive Injection

SM90 nvcc (CUDA 12.9 AND 13.0) injects `warpgroup.arrive` barriers for ALL warp groups
in any CTA containing WGMMA instructions. If WG3 takes a different code path (non-WGMMA),
it never reaches the barrier injection points → **DEADLOCK**.

### Evidence
1. Simple 4-WG test (no WGMMA): works ✓
2. FA4 4-WG with WG3 exit/return: deadlocks ✗
3. FA4 4-WG without setmaxnreg: still deadlocks ✗
4. FA4 4-WG with WG3 spin-wait (no return): still deadlocks ✗

### Why AVO Blackwell Works
Blackwell SM120 compiler likely only injects warpgroup.arrive for WGs that execute WGMMA.
Non-WGMMA WGs (softmax, correction, load) can safely take different code paths.

## Viable 4-WG Design: BLOCK_Q=192 with 3 Consumers

All WGs execute the SAME consumer code path (including WGMMA). Compiler barriers satisfied.

### Architecture
- WG0 (128 threads): Producer — TMA loads Q/K/V
- WG1 (128 threads): Consumer — Q rows 0-63
- WG2 (128 threads): Consumer — Q rows 64-127
- WG3 (128 threads): Consumer — Q rows 128-191
- BLOCK_Q = 192, BLOCK_KV = 128, DIM = 128

### Register Budget
- 512 threads × 128 regs = 65,536 ✓
- IS_CAUSAL template reduces consumer code to ≤ 128 regs (0 spills confirmed)
- No setmaxnreg needed (all threads start at 128, producer uses fewer naturally)

Wait — setmaxnreg IS needed because producer WG does not execute WGMMA.
Without setmaxnreg, producer at 128 regs and consumers at 128 regs = total 65536. ✓
But can consumers do m64n128k16 QK (64 output regs) + PV (64 output regs) in 128 regs?

With IS_CAUSAL=true template: ptxas confirms 128 regs, 0 spills. YES! ✓

### SMEM Layout
- Q_lo: 192 × 64 × 2 = 24,576 bytes (24KB)
- Q_hi: 24KB
- K ×2: 64KB
- V ×2: 64KB
- Total: 48 + 64 + 64 = 176KB + barriers < 232KB ✓

### Q TMA Loading
Box [64, 64]: 3 loads per Q half (rows 0-63, 64-127, 128-191)
6 TMA calls total for Q (vs 2 in 3-WG)

### Inter-WG Barriers
3 consumer WGs: use bar.sync with 384 threads (simpler than asymmetric 2-WG pattern)

### Changes Required
1. BLOCK_Q: 128 → 192
2. Q SMEM: 32KB → 48KB
3. Q TMA box: [64, 128] → [64, 64], 6 loads instead of 2
4. Consumer WG count: 2 → 3 (cwg = 0, 1, 2)
5. Inter-WG barriers: bar.sync 384 threads
6. Block count: ceil(S/192) instead of ceil(S/128)
7. Causal max_kv_iter: adjust for BLOCK_Q=192
8. Epilogue: 384 consumer threads + TMA S2G

### Risks
- BLOCK_Q=192 not power-of-2 (may affect Q tile boundary handling)
- Fewer blocks per head → less parallelism for small S
- 3-consumer barrier coordination more complex
- Performance may not improve if 2× QK overhead (n=64 sub-tiles) dominates

### Alternative: Keep BLOCK_Q=128, WG3 = Shadow Consumer
WG3 executes the same consumer code as WG1 (same Q rows 0-63) but writes O to /dev/null.
Pro: minimal code change, all barriers satisfied.
Con: 25% wasted compute (WG3 does duplicate work).
