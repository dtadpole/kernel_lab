## Implementation Techniques

### Reference: CuTe DSL (`conf/fixtures/matmul/cute_gemm.py`)

NVIDIA CUTLASS CuTe DSL with MLIR JIT compilation. Matches the official
`Sm120GemmKernel` architecture from CUTLASS examples.

| Parameter | Value |
|-----------|-------|
| Tile shape | 128×128×64 (M×N×K) |
| Threads | 160 (4 MMA warps + 1 DMA warp) |
| Pipeline | 3-stage PipelineTmaAsync |
| Registers | 207/thread, 0 spills |
| SMEM | 99.3 KB (3 × 32KB/stage + mbarrier) |

**Techniques:**
1. **TMA (Tensor Memory Accelerator)** — `cp.async.bulk.tensor.2d` for global→SMEM, hardware-driven 2D tile copies
2. **Warp specialization** — 1 DMA warp produces data, 4 MMA warps consume. Decouples memory and compute pipelines
3. **PipelineTmaAsync** — 3-stage SMEM pipeline with mbarrier synchronization for producer-consumer overlap
4. **ldmatrix S2R** — `ldmatrix.sync.aligned.x4.m8n8` feeds MMA operands directly from SMEM
5. **mma.sync m16n8k16** — BF16 Tensor Core with FP32 accumulation, 2×2×1 warp atom layout
6. **Persistent tile scheduling** — `StaticPersistentTileScheduler` with 188 CTAs (1 per SM), work loop for multiple tiles
7. **CuTe tensor caching** — Reuses `from_dlpack` wrappers across calls when input pointers unchanged, eliminates 17μs Python dispatch overhead
8. **BF16 epilogue** — FP32→BF16 conversion in-kernel, pre-allocated output buffer

### Generated: Hand-Written CUDA PTX (`data/generated/sm120/matmul/generated.cu`)

Two size-specialized kernel variants with inline PTX assembly.

| Parameter | Big (M,N ≥ 2048) | Small (M,N < 2048) |
|-----------|-------------------|---------------------|
| Tile shape | 256×128×32 | 128×64×32 |
| Threads | 256 (8 warps) | 128 (4 warps) |
| Pipeline | 3-stage | 4-stage |
| Registers | 191, 0 spills | 134, 0 spills |
| SMEM | 73.9 KB | 49.3 KB |

**Techniques:**
1. **TMA via inline PTX** — `cp.async.bulk.tensor.2d.shared::cta.global.tile` with explicit `CUtensorMap` encoding through driver API
2. **Manual mbarrier pipeline** — `mbarrier.init / arrive.expect_tx / try_wait.parity` for producer-consumer sync with phase tracking
3. **ldmatrix** — `ldmatrix.sync.aligned.x4.m8n8.shared.b16` (A) and `ldmatrix.sync.aligned.x2.trans.m8n8.shared.b16` (B transposed)
4. **mma.sync m16n8k16** — Same Tensor Core op as reference, but all 8 warps do MMA (no dedicated DMA warp)
5. **SMEM swizzle** — A uses 64B swizzle, B uses 128B swizzle to avoid bank conflicts
6. **Interleaved load+compute** — A-tile loads interleaved with MMA in groups of 2 M-sub-tiles
7. **Persistent grid** — `min(totalTiles, numSMs)` blocks with tile loop, 4×4 super-tile ordering for L2 locality
8. **Multi-variant dispatch** — Runtime size-based selection between big and small kernels
9. **BF16 epilogue** — Vectorized `__floats2bfloat162_rn` stores (2 elements per memory transaction)
10. **`__launch_bounds__(N, 1)`** — Explicit occupancy hint for maximum register budget

## Lessons Learned

### 1. NCU data drives decisions, not intuition

Every optimization hypothesis was tested against NCU data before implementation:
- **TILE_K=64 hypothesis** (reduce K-loop overhead) → NCU revealed mbarrier 3-TMA limit, saving days of debugging
- **Non-persistent scheduling hypothesis** (cuBLAS uses 2.72 waves) → benchmark showed 0% improvement, persistent tile loop overhead is negligible
- **CuTe DSL 256×128 tile hypothesis** → NCU showed 255 regs/14% compute, immediately explained by accumulator size calculation
- **Warp stall analysis** → `math_pipe_throttle` 7.64 vs cuBLAS 5.11 identified the exact remaining optimization target

### 2. Hardware limits are discoverable through experiments

Three undocumented hardware constraints found through testing:
- **mbarrier max 3 TMA completions per phase** — >3 TMA loads per mbarrier causes deadlock or crash
- **TMA box dim0 limit** — 32 elements with 64B swizzle, 64 with 128B swizzle
- **CuTe DSL atom_layout constrains warp count** — atom_layout product must equal num_mma_warps; mismatch causes illegal memory access

### 3. Register pressure is the key constraint for tile size

The 128×128 vs 256×128 tile choice is entirely determined by register budget:
- 128×128 with 4 warps: 64 float accumulator regs/warp → 207 total regs ✓
- 256×128 with 4 warps: 128 float accumulator regs/warp → 255 regs + spill → 6x slower
- 256×128 with 8 warps: 64 float accumulator regs/warp → but `setmaxregister` applied globally → 40 regs
- Hand-written PTX can control register allocation precisely, enabling 256×128 with 191 regs and 0 spills

### 4. Python dispatch overhead matters for small matrices

The CuTe DSL reference improved 2.24x at 256×256 just from caching CuTe tensor wrappers (~17μs saved). At small sizes where kernel execution is <10μs, dispatch overhead dominates. At large sizes (8192), the same optimization gives only 1% because kernel time (2.7ms) dwarfs the overhead.

### 5. The last 5% requires instruction-level optimization

Generated kernel reaches 88.6% of peak (446 TFLOPS) with macro-level optimizations (tile shape, pipeline, TMA). The remaining gap to cuBLAS (86.7% compute vs 85.1%) comes from instruction scheduling: cuBLAS has 28% less stall per instruction (11.50 vs 15.98 cycles). The top stall source is `math_pipe_throttle` — MMA instructions are too back-to-back, need better ldmatrix interleaving.

### 6. Warp specialization is a trade-off, not always a win

CuTe DSL's warp specialization (1 DMA + 4 MMA warps) gives clean producer-consumer separation but wastes 1 warp on DMA duties and limits occupancy to 10.4%. The generated kernel uses all 8 warps for MMA (thread 0 issues TMA with 1 PTX instruction) and achieves 16.7% occupancy — matching cuBLAS.

## Next Steps

The NCU warp stall analysis points to one clear optimization target:
- **Improve ldmatrix/MMA interleaving** in the generated kernel's K-loop inner body to reduce `math_pipe_throttle` from 7.64 → closer to cuBLAS's 5.11
