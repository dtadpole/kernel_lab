# FA4 WGMMA + RS Mode + Double-Buffered K Pipeline

## Hardware

- **GPU:** NVIDIA H100 SXM5 80GB HBM3
- **Architecture:** SM 9.0a (Hopper)
- **SMs:** 132
- **Peak BF16 Tensor Core:** 989.5 TFLOPS (dense)
- **HBM3 Bandwidth:** 3,352 GB/s
- **SMEM per SM:** 228 KB
- **Host:** devvm8490.cco0.facebook.com (h8_3), GPU 4
- **CUDA Toolkit:** 12.8

## Objective

Rewrite the FA4 forward kernel from SM80-compatible `mma.sync.m16n8k16` to
SM90-native WGMMA (`wgmma.mma_async.sync.aligned`) to unlock async MMA
execution, eliminate ldmatrix overhead, and enable software pipelining.

## Approach

Phased rewrite of `data/generated/sm90/fa4/generated.cu`:

1. **Phase 1 — WGMMA compute:** Replace mma.sync with WGMMA m64n64k16 (QK)
   and m64n128k16 (PV). SS mode for both. Unified 128 threads, no pipeline.
   Result: 248–332 TFLOPS (marginal improvement — no async overlap realized).

2. **Phase 2 — Warp specialization (reverted):** Split into 32-thread producer
   + 128-thread consumer with named barriers. Result: 237–280 TFLOPS
   (REGRESSION — 32 threads too slow for cp.async loads). Reverted.

3. **Phase 3 — Software-pipelined double-buffered K:** Keep 128 unified threads.
   Double-buffer K: K[i+1] loads via cp.async overlap with QK[i] + softmax.
   Added dedicated V buffer for V load overlap. Result: 374–449 TFLOPS.

4. **Phase 4 — RS mode PV GEMM:** Eliminate P SMEM roundtrip. After softmax,
   P stays in registers. PV uses WGMMA RS mode (A from registers, B from SMEM
   descriptor). Removes 8KB SMEM store + fence + sync from hot loop.
   Result: 365–448 TFLOPS.

## Kernel Architecture

| Parameter | Value |
|-----------|-------|
| Threads | 128 (1 warp group, 4 warps) |
| tile_m (BLOCK_Q) | 64 |
| tile_n (BLOCK_KV) | 64 |
| DIM (head_dim) | 128 |
| QK WGMMA | `m64n64k16.f32.bf16.bf16` (SS mode, tnspA=0, tnspB=0) |
| PV WGMMA | `m64n128k16.f32.bf16.bf16` (RS mode, tnspB=1) |
| Pipeline | 2-stage K (double-buffered), dedicated V buffer |
| SMEM | Q(16KB) + K[0](16KB) + K[1](16KB) + V(16KB) = 64KB |
| Registers | 168/thread, 0 spills |
| Occupancy | 1 block/SM |

### Key Optimizations

- **WGMMA async execution:** QK GEMM runs async while cp.async K[i+1] loads
- **RS mode PV:** Softmax output packed in registers via `cvt.rn.bf16x2.f32`,
  fed directly to WGMMA RS mode — no P SMEM store/load cycle
- **B128 swizzle layout:** `Swizzle<3,4,3>` for all SMEM tiles, matching
  WGMMA descriptor SWIZZLE_128B format
- **Descriptor K-stepping:** `gmma_desc_advance()` adds to start_address
  (base_offset always 0, validated from matmul kernel)
- **V K-stepping for tnspB=1:** Advance by `2 × stride / 16 = 256` per K-step
  (16 rows = 2 core matrix groups), not column-wise `ks * 2`

|--------|-------------------|-----------------|---------|
| causal-b8-s4096 | 250.7 | 364.7 | 1.45× |
| causal-b4-s8192 | 271.0 | 395.9 | 1.46× |
| causal-b2-s16384 | 280.1 | 411.4 | 1.47× |
| nc-b8-s4096 | 277.9 | 426.8 | 1.54× |
| nc-b4-s8192 | 291.7 | 444.7 | 1.52× |
| nc-b2-s16384 | 298.4 | 447.5 | 1.50× |

**Average speedup: 1.49×** over mma.sync baseline.

## What Worked

1. **WGMMA + double-buffered K pipeline** (Phase 3) — biggest single gain.
   K[i+1] loading overlaps with QK[i] compute + softmax. All 128 threads
   cooperate on both loads and compute (no warp specialization).

2. **RS mode PV GEMM** (Phase 4) — eliminates P SMEM roundtrip. Softmax
   output packed directly into WGMMA A operand registers.

3. **B128 swizzle layout** — compatible with WGMMA SWIZZLE_128B descriptor.
   Atom-based layout (64 BF16 cols per atom) matches WGMMA's 128B access.

## What Didn't Work

1. **Warp specialization with cp.async** (Phase 2) — 32-thread producer
   loads 4× slower than 128-thread unified loading. The pipeline overlap
   doesn't compensate. TMA (hardware loads) would fix this, but requires
   per-(batch,head) descriptor setup.

2. **tile_n=128** — doubling KV tile halves iterations but doubles per-iter
   cost. Net neutral (0.96–1.00×). Would benefit more with TMA or larger
   occupancy.

## Remaining Gap to CuTe DSL (~1.6×)

| Gap Source | Impact | Fix |
|-----------|--------|-----|
| cp.async vs TMA | ~1.2× | TMA loads (hardware tensor movement) |
| 1 WG vs 2 WGs | ~1.1× | 2 MMA warp groups with intra-WG overlap |
| tile 64×64 vs 128×128 | ~1.1× | Larger tiles (requires more SMEM/regs) |
| Address computation | ~1.05× | TMA eliminates swizzle address calc |

## Known Issue

Correctness bug at output columns 120-127 (last 8-col chunk) on pathological
arange test data (values ~8000 in BF16). Max error = 960 (11.7% relative).
Other columns are correct within BF16 tolerance (0–64 error, <0.8% relative).
Root cause: V descriptor atom boundary handling with tnspB=1 in B128 swizzle.
Needs investigation — may require adjusting the V descriptor stride or
leading_byte_offset for transposed access patterns.
