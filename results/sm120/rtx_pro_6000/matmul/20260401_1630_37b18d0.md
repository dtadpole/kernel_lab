## Implementation Techniques

### Reference: CuTe DSL (`conf/fixtures/matmul/cute_gemm.py`)

The reference uses NVIDIA's CuTe DSL (CUTLASS Python bindings) with MLIR JIT compilation.

**Architecture:**
- **Tile shape**: 128Ă—128Ă—64 (MĂ—NĂ—K)
- **Thread count**: 160 (5 warps: 1 DMA + 4 MMA)
- **Pipeline stages**: 3
- **Register budget**: ~232 (MMA warps), ~40 (DMA warp) via `setmaxregister`

**Key techniques:**
1. **TMA (Tensor Memory Accelerator)** â€” `cp.async.bulk.tensor.2d` for globalâ†’SMEM loads. Hardware-driven 2D tile copies with zero thread involvement. Uses `CopyBulkTensorTileG2SOp`.
2. **Warp specialization** â€” 1 DMA warp (producer) handles all TMA loads; 4 MMA warps (consumers) do compute. Decouples memory and compute for better overlap.
3. **PipelineTmaAsync** â€” Multi-stage (3-stage) software pipeline with `mbarrier` synchronization. Producer acquires/commits stages; consumers wait/release.
4. **ldmatrix S2R** â€” `ldmatrix.sync.aligned.x4.m8n8` for sharedâ†’register data movement, directly feeding MMA operands.
5. **mma.sync** â€” `mma.sync.aligned.m16n8k16.f32.bf16.bf16.f32` Tensor Core operation with 2Ă—2Ă—1 warp layout (4 warps Ă— 32Ă—16 per warp = 128Ă—128 tile).
6. **Persistent tile scheduling** â€” `StaticPersistentTileScheduler` assigns multiple output tiles per CTA via work loop. Max 188 CTAs (matching SM count).
7. **Zero-copy input** â€” `from_dlpack(B.t())` creates transposed view without copying data. B layout: (N,K) N-major from PyTorch's column-major view.
8. **BF16 epilogue** â€” FP32 accumulator â†’ BF16 output conversion for reduced memory traffic.

### Generated: Hand-Written CUDA (`data/generated/sm120/matmul/generated.cu`)

The generated kernel is hand-written PTX-level CUDA with two size-specialized variants.

**Architecture (Big kernel, M,N â‰Ą 2048):**
- **Tile shape**: 256Ă—128Ă—32 (MĂ—NĂ—K)
- **Thread count**: 256 (8 warps: 2MĂ—4N layout)
- **Pipeline stages**: 3
- **Registers**: 191 (0 spills)

**Architecture (Small kernel, M,N < 2048):**
- **Tile shape**: 128Ă—64Ă—32 (MĂ—NĂ—K)
- **Thread count**: 128 (4 warps: 2MĂ—2N layout)
- **Pipeline stages**: 4
- **Registers**: 134 (0 spills)

**Key techniques:**
1. **TMA via inline PTX** â€” `cp.async.bulk.tensor.2d.shared::cta.global.tile` with explicit `CUtensorMap` encoding via driver API (`cuTensorMapEncodeTiled`).
2. **mbarrier pipeline** â€” Manual `mbarrier.init` / `mbarrier.arrive.expect_tx` / `mbarrier.try_wait.parity` for producer-consumer synchronization. Phase-based completion tracking.
3. **ldmatrix** â€” `ldmatrix.sync.aligned.x4.m8n8.shared.b16` for A tiles; `ldmatrix.sync.aligned.x2.trans.m8n8.shared.b16` for B tiles (transposed load).
4. **mma.sync** â€” Same `mma.sync.aligned.m16n8k16.row.col.f32.bf16.bf16.f32` as reference.
5. **SMEM swizzle** â€” A uses 64B swizzle (`CU_TENSOR_MAP_SWIZZLE_64B`), B uses 128B swizzle (`CU_TENSOR_MAP_SWIZZLE_128B`) to avoid bank conflicts.
6. **Interleaved load + compute** â€” Big kernel interleaves A-tile loads with MMA in groups of 2, overlapping ldmatrix latency with compute.
7. **Persistent grid** â€” `min(totalTiles, numSMs)` blocks with tile loop. 4Ă—4 super-tile ordering for L2 locality when grid is large enough.
8. **Multi-tile dispatch** â€” Size-based kernel selection at launch time. Big kernel has wider tiles (256Ă—128) for better compute density at large sizes; small kernel has more pipeline stages (4) for better latency hiding at small sizes.
9. **BF16 epilogue** â€” Direct `__floats2bfloat162_rn` vectorized stores (bfloat162 = 2 elements per store).
10. **`__launch_bounds__(N, 1)`** â€” Explicit occupancy hint, 1 block per SM for maximum register budget.

## Key Observations

### Performance

1. **Generated beats cuBLAS at small/medium sizes** (256â€“2048): up to 2.48x at 256Ă—256. The generated kernel's multi-tile dispatch with size-specific tile shapes gives it an advantage where cuBLAS's large-tile heuristics have too much overhead.

2. **Generated matches cuBLAS at large sizes** (4096â€“8192): within 1% at 4096, tied at 8192. Both achieve ~88% of peak, suggesting the problem is truly compute-bound at these sizes.

3. **Reference is slower due to Python/MLIR overhead**: The CuTe DSL kernel has ~17ÎĽs per-call overhead from the Pythonâ†’MLIRâ†’CUDA dispatch path. This dominates at small sizes (256â€“512) where kernel execution itself is <10ÎĽs. At 8192Ă—8192 the reference reaches 80% peak â€” the GPU kernel itself is competitive, but the dispatch overhead caps it below the other two.

4. **No register spills in generated kernel**: 191 registers (big) and 134 registers (small), both with 0 spills. The reference uses up to 207 registers (NCU measured).

### NCU Insights

5. **cuBLAS launches 512 blocks (2.72 waves/SM)** vs our 188 (1.00 wave). When a tile finishes, the next block starts immediately. Our persistent-grid approach (1 block/SM with tile loop) has less scheduling flexibility â€” the HW warp scheduler can't interleave warps from different blocks.

6. **Reference has high L2/memory throughput (71%)** but lower compute (84%) â€” its 128Ă—128 tile produces smaller output tiles and accesses SMEM more frequently per FLOP. Its 5 warp/SM (vs 8 for generated) means less latency hiding.

7. **cuBLAS is CUTLASS-based**: The kernel name `cutlass_80_tensorop_bf16_s16816gemm_relu_bf16_256x128_32x3_nn_align8` reveals it uses 256Ă—128 tile, 32 K-tile, 3-stage pipeline â€” the same architecture as our generated kernel, but with higher register budget (218 vs 191) and multi-wave scheduling.

### Improvement Opportunities

8. **Multi-wave scheduling**: Launch more blocks than SMs (like cuBLAS's 512). This is the biggest gap â€” our 1 wave/SM vs cuBLAS's 2.72 waves means idle SM time between tile transitions in the persistent loop.

9. **Consider 128Ă—128 tile + 2 blocks/SM**: This would double occupancy (32% theoretical) and give the warp scheduler more warps to hide latency, at the cost of smaller tiles.

10. **Wider TILE_K** (64 instead of 32): Would reduce K-loop iterations and amortize pipeline/barrier overhead. Reference already uses bK=64.
